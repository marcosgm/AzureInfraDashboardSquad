## Design Review Decisions

### 1. Azure SDK Selection: `@azure/arm-resources`
- **Choice:** Use `ResourceManagementClient(credential, subscriptionId).resources.list()`
- **Rationale:** Correct SDK for enumerating all resources in a subscription across resource groups
- **Traps Avoided:** Not `@azure/arm-resources-subscriptions` (subscription listing) or `@azure/arm-subscriptions` v6 (lifecycle management)

### 2. Data Model (New Types in `src/types/azure.ts`)
```typescript
interface AzureResource {
  id: string;
  name: string;
  type: string;
  resourceGroup: string;  // Parsed from ID
  location: string;
}

interface ResourceRelationship {
  sourceId: string;
  targetId: string;
  type: string;  // "contains", "links", "depends-on", etc.
}

interface ResourceGraph {
  nodes: AzureResource[];
  edges: ResourceRelationship[];
}

interface ResourceGraphResponse {
  subscriptionId: string;
  resourceGraph: ResourceGraph;
  cachedAt: string;
}
```

### 3. Backend Architecture
- **Service:** `src/services/azure/resources.ts` â€” lists resources via `ResourceManagementClient`

### 4. Relationship Discovery Strategy (MVP)
**No Azure Resource Graph queries.** Use:
- **Parent/child from ID containment:** VNet ID contains Subnet ID
- **Hardcoded type mapping:** 
  - VM â†’ NIC (via `networkProfile.networkInterfaces`)
  - NIC â†’ Subnet (via `ipConfigurations[0].subnet.id`)
  - WebApp â†’ AppServicePlan (via `appServicePlanId`)
- **Result:** Pure function in `relationships.ts` takes resource list, outputs edges

### 5. Diagram Library: React Flow + Dagre
- **Canvas:** `@xyflow/react` (React Flow v12, MIT license, no Pro required)
- **Auto-Layout:** `@dagrejs/dagre` (MIT, hierarchical layout algorithm)
- **Rationale:** 
  - React Flow has 45K+ GitHub stars, mature ecosystem
  - Dagre is battle-tested for directed graphs
  - Combined cost: zero (both free, no subscription)
